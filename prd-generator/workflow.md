# PRD Generator 工作流程

## 完整工作流程图

```
用户请求生成PRD
        ↓
    Step 1: 页面分析
    ├─ 读取 index.tsx
    ├─ 读取 spec.md
    ├─ 读取 content.md
    └─ 查看 screenshot.png
        ↓
    Step 2: 类型识别
    ├─ 数据看板类？ ──是──→ 使用 data-prd-template.md
    ├─ 后台列表类？ ──是──→ 使用 backend-list-prd-template.md
    ├─ 表单编辑类？ ──是──→ 使用 backend-form-edit-prd-template.md
    ├─ 表单预览类？ ──是──→ 使用 backend-form-preview-prd-template.md
    └─ 用户前台类？ ──是──→ 使用 user-frontend-prd-template.md
        ↓
    Step 3: 需求确认 (可选)
    ├─ 确认页面类型
    ├─ 确认功能范围
    └─ 确认特殊要求
        ↓
    Step 4: PRD生成
    ├─ 应用对应模板
    ├─ 提取页面信息
    ├─ 填充模板内容
    └─ 保存到 src/pages/<page-name>/prd.md
        ↓
    Step 5: 一致性核对
    ├─ 核对功能点与页面实现
    ├─ 核对技术约定与spec.md
    ├─ 核对范围与用户要求
    └─ 核对是否遗漏核心内容
        ↓
    一致性检查通过？
        ├─ 是 → 完成！交付PRD
        └─ 否 → 返回 Step 4 重新生成
```

## 详细步骤说明

### Step 1: 页面分析

**目标**：全面了解页面的功能、结构和技术实现。

**操作清单**：

- [ ] 读取 `index.tsx`（或主要实现文件）
- [ ] 读取 `spec.md` 规格文档（如果存在）
- [ ] 读取 `content.md` 或其他相关文档
- [ ] 查看 `screenshot.png` 了解页面视觉
- [ ] 识别页面的主要功能模块
- [ ] 记录关键技术实现细节

**关键关注点**：

- 页面包含哪些组件
- 有哪些交互逻辑
- 使用了哪些第三方库
- 数据结构是怎样的

***

### Step 2: 类型识别

**目标**：自动识别页面类型，选择最合适的模板。

**识别规则**：

| 判断条件               | 页面类型    | 模板文件                                 |
| ------------------ | ------- | ------------------------------------ |
| 包含大量图表、指标卡片、数据看板   | 数据看板类   | data-prd-template.md                 |
| 包含数据列表、筛选器、分页、批量操作 | 后台列表类   | backend-list-prd-template.md         |
| 包含表单输入、提交按钮、字段校验   | 后台表单编辑类 | backend-form-edit-prd-template.md    |
| 只有数据展示、无输入操作       | 后台表单预览类 | backend-form-preview-prd-template.md |
| C端用户界面、营销活动、产品展示   | 用户前台类   | user-frontend-prd-template.md        |

**如果无法自动识别**：

- 向用户询问页面类型
- 提供5种类型供用户选择

***

### Step 3: 需求确认 (可选)

**目标**：确保生成的PRD符合用户期望。

**确认清单**：

- [ ] 确认页面类型（如未自动识别）
- [ ] 确认是否有指定的功能范围
- [ ] 确认是否有特殊要求或关注点
- [ ] 确认PRD的详细程度

**向用户确认的问题示例**：

```
我已分析了页面内容，识别为[数据看板类]页面。

请问：
1. 页面类型是否正确？
2. 是否需要指定生成某个特定功能范围的PRD？
3. 有什么特殊要求或关注点吗？
```

***

### Step 4: PRD生成

**目标**：根据模板生成结构化的PRD文档。

**生成流程**：

1. **读取模板**
   - 根据页面类型加载对应模板
   - 理解模板的结构和要求
2. **信息提取**
   - 从 `index.tsx` 提取功能点
   - 从 `spec.md` 提取技术约定
   - 从 `content.md` 提取文案内容
   - 从 `screenshot.png` 提取布局信息
3. **填充模板**
   - 替换模板中的占位符
   - 按照模板结构组织内容
   - 确保术语统一、逻辑清晰
4. **保存文件**
   - 路径：`src/prototypes/<page-name>/prd.md`
   - 文件名：`prd.md`（小写）
   - 确保路径和文件名正确

**关键原则**：

- 内容必须基于实际页面实现
- 使用研发/测试通用术语
- 避免冗余表述
- 结构清晰、层次分明

***

### Step 5: 一致性核对

**目标**：确保生成的PRD准确无误。

**核对维度**：

#### 维度1: 功能点与页面实现一致

- [ ] PRD中描述的功能点在页面中都存在
- [ ] 没有描述页面中不存在的功能
- [ ] 交互逻辑描述与实现一致
- [ ] 展示规则描述与实现一致

#### 维度2: 技术约定与spec.md一致

- [ ] 数据口径定义与spec.md一致
- [ ] 字段格式定义与spec.md一致
- [ ] 接口规则定义与spec.md一致
- [ ] 没有与spec.md冲突的内容

#### 维度3: 范围与用户要求一致

- [ ] 如果用户指定了范围，只包含范围内的内容
- [ ] 没有超出用户指定的范围
- [ ] 没有遗漏用户指定范围内的核心内容

#### 维度4: 无遗漏核心内容

- [ ] 所有主要功能模块都已描述
- [ ] 所有关键字段都已定义
- [ ] 所有重要交互都已说明
- [ ] 模板的所有必填部分都已填充

**核对结果处理**：

- ✅ 全部通过：完成！交付PRD
- ❌ 发现问题：返回Step 4重新生成，修正问题后再次核对

***

## 常见问题处理

### Q: 页面同时包含多种类型的功能怎么办？

A: 选择最主要的功能类型作为主模板，其他类型的功能可以在对应模块中详细描述。

### Q: spec.md不存在怎么办？

A: 只基于页面实现生成PRD，在PRD中注明"规格文档待补充"。

### Q: 用户要求修改已生成的PRD怎么办？

A: 了解用户的具体修改需求，重新执行Step 4和Step 5。

### Q: 页面非常复杂，一个模板不够用怎么办？

A: 可以将页面拆分为多个模块，每个模块使用合适的子模板结构。

***

## 最佳实践

1. **先分析再生成**：充分理解页面后再开始生成
2. **保持客观**：只描述实际存在的功能，不添加主观臆测
3. **注重细节**：字段定义、校验规则等细节要准确
4. **核对再交付**：一定要经过一致性核对后再交付
5. **持续改进**：根据用户反馈不断优化生成质量
